# 数据库核心表设计草案（MVP 版）

## 一、设计原则

首月数据库设计遵守 4 个原则：

1. 先满足 MVP，不追求过度抽象
2. 用关系型数据库表达知识图，不首月引入图数据库
3. AI 原始结果必须留痕
4. 错题、笔记、标签、审核、解题记录必须能关联起来

---

## 二、核心实体总览

首月建议至少包含以下表：

1. `users`
2. `files`
3. `notes`
4. `problems`
5. `knowledge_tags`
6. `note_tag_links`
7. `problem_tag_links`
8. `tag_relations`
9. `parse_jobs`
10. `review_tasks`
11. `knowledge_mastery`
12. `question_bank_items`
13. `ai_solution_records`

---

## 三、表设计建议

## 1. users

用于用户信息。

建议字段：

- `id` UUID PK
- `username` varchar unique
- `email` varchar nullable unique
- `password_hash` varchar
- `status` varchar default 'active'
- `created_at` timestamp
- `updated_at` timestamp

---

## 2. files

用于文件元数据记录。

建议字段：

- `id` UUID PK
- `user_id` UUID FK -> users.id
- `object_key` varchar unique
- `storage_provider` varchar
- `bucket_name` varchar
- `file_url` text
- `original_filename` varchar
- `mime_type` varchar
- `size_bytes` bigint
- `sha256` varchar
- `source_type` varchar default 'upload'
- `upload_status` varchar
- `created_at` timestamp
- `updated_at` timestamp

建议索引：

- `(user_id, created_at desc)`
- `(user_id, sha256)`
- `object_key unique`

---

## 3. notes

用于笔记实体。

建议字段：

- `id` UUID PK
- `user_id` UUID FK
- `file_id` UUID FK nullable
- `title` varchar
- `subject` varchar
- `raw_text` text
- `cleaned_text` text
- `parse_status` varchar default 'pending'
- `source_type` varchar default 'upload'
- `created_at` timestamp
- `updated_at` timestamp

可选增强：

- `search_vector` tsvector
- `extra_json` jsonb

---

## 4. problems

用于错题/题目实体。

建议字段：

- `id` UUID PK
- `user_id` UUID FK
- `file_id` UUID FK nullable
- `title` varchar nullable
- `subject` varchar
- `stem_text` text
- `normalized_stem` text
- `answer_text` text nullable
- `explanation_text` text nullable
- `difficulty` varchar nullable
- `question_type` varchar nullable
- `parse_status` varchar default 'pending'
- `source_type` varchar default 'upload'
- `created_at` timestamp
- `updated_at` timestamp

建议索引：

- `(user_id, subject, created_at desc)`
- `GIN(normalized_stem)` 或全文索引

---

## 5. knowledge_tags

规范知识点表，是整个系统的核心枢纽。

建议字段：

- `id` UUID PK
- `subject` varchar
- `canonical_name` varchar
- `display_name` varchar
- `description` text nullable
- `level` int nullable
- `parent_id` UUID FK nullable -> knowledge_tags.id
- `aliases_json` jsonb nullable
- `status` varchar default 'active'
- `created_at` timestamp
- `updated_at` timestamp

建议约束：

- `(subject, canonical_name)` 唯一

---

## 6. note_tag_links

笔记与知识点关联表。

建议字段：

- `id` UUID PK
- `note_id` UUID FK -> notes.id
- `tag_id` UUID FK -> knowledge_tags.id
- `confidence` numeric(4,3)
- `source` varchar default 'llm'
- `status` varchar default 'pending'
- `created_at` timestamp
- `updated_at` timestamp

状态建议：

- `pending`
- `confirmed`
- `rejected`

---

## 7. problem_tag_links

错题与知识点关联表。

建议字段：

- `id` UUID PK
- `problem_id` UUID FK -> problems.id
- `tag_id` UUID FK -> knowledge_tags.id
- `confidence` numeric(4,3)
- `source` varchar default 'llm'
- `status` varchar default 'pending'
- `created_at` timestamp
- `updated_at` timestamp

---

## 8. tag_relations

用于表达标签之间的关系，形成“轻量知识图”的边。

建议字段：

- `id` UUID PK
- `user_id` UUID FK nullable
- `from_tag_id` UUID FK -> knowledge_tags.id
- `to_tag_id` UUID FK -> knowledge_tags.id
- `relation_type` varchar
- `weight` numeric(8,3) default 1
- `created_at` timestamp
- `updated_at` timestamp

首月 `relation_type` 可只支持：

- `cooccur`
- `prerequisite`（可选，允许为空）

---

## 9. parse_jobs

用于 AI 解析任务。

建议字段：

- `id` UUID PK
- `user_id` UUID FK
- `entity_type` varchar  -- note / problem
- `entity_id` UUID
- `file_id` UUID FK nullable
- `status` varchar default 'pending'
- `attempt_count` int default 0
- `llm_model` varchar
- `request_payload_json` jsonb
- `result_json` jsonb nullable
- `error_message` text nullable
- `started_at` timestamp nullable
- `finished_at` timestamp nullable
- `created_at` timestamp
- `updated_at` timestamp

状态建议：

- `pending`
- `running`
- `success`
- `failed`

---

## 10. review_tasks

用于人工审核。

建议字段：

- `id` UUID PK
- `user_id` UUID FK
- `task_type` varchar   -- tag_review / parse_review / solve_review
- `entity_type` varchar -- note / problem / solution
- `entity_id` UUID
- `payload_json` jsonb
- `status` varchar default 'pending'
- `created_at` timestamp
- `updated_at` timestamp

状态建议：

- `pending`
- `approved`
- `rejected`
- `fixed`

---

## 11. knowledge_mastery

用于用户在知识点上的学习状态统计。

建议字段：

- `id` UUID PK
- `user_id` UUID FK
- `tag_id` UUID FK
- `total_problem_count` int default 0
- `wrong_count` int default 0
- `right_count` int default 0
- `note_count` int default 0
- `mastery_score` numeric(5,2) default 0
- `last_wrong_at` timestamp nullable
- `updated_at` timestamp

建议约束：

- `(user_id, tag_id)` 唯一

---

## 12. question_bank_items

本地题库表。

建议字段：

- `id` UUID PK
- `subject` varchar
- `title` varchar nullable
- `stem_text` text
- `normalized_stem` text
- `answer_text` text nullable
- `explanation_text` text nullable
- `tags_json` jsonb nullable
- `source` varchar default 'manual'
- `status` varchar default 'active'
- `created_at` timestamp
- `updated_at` timestamp

说明：

首月可以不做复杂导题平台，但要留好表结构。

---

## 13. ai_solution_records

用于记录本地题库未命中后的 AI 参考解析。

建议字段：

- `id` UUID PK
- `user_id` UUID FK
- `problem_id` UUID FK nullable
- `problem_hash` varchar
- `original_question_text` text
- `normalized_question_text` text
- `local_search_status` varchar
- `local_top_score` numeric(6,3) nullable
- `solver_model` varchar
- `verifier_model` varchar nullable
- `final_answer` text
- `solution_steps_json` jsonb
- `knowledge_points_json` jsonb
- `confidence` numeric(4,3)
- `warning_text` text
- `created_at` timestamp

---

## 四、推荐关系图（文字版）

```text
users
  ├─ files
  ├─ notes
  ├─ problems
  ├─ parse_jobs
  ├─ review_tasks
  ├─ knowledge_mastery
  └─ ai_solution_records

notes
  └─ note_tag_links ── knowledge_tags

problems
  └─ problem_tag_links ── knowledge_tags

knowledge_tags
  └─ tag_relations (from_tag_id / to_tag_id)

problems
  └─ ai_solution_records

question_bank_items
  -> 本地搜索与解题命中源
```

---

## 五、首月建议优先实现的索引

1. `files(user_id, created_at desc)`
2. `notes(user_id, subject, created_at desc)`
3. `problems(user_id, subject, created_at desc)`
4. `knowledge_tags(subject, canonical_name unique)`
5. `note_tag_links(note_id, tag_id)`
6. `problem_tag_links(problem_id, tag_id)`
7. `knowledge_mastery(user_id, tag_id unique)`
8. `parse_jobs(status, created_at)`
9. `review_tasks(status, created_at)`
10. `question_bank_items` 的全文索引

---

## 六、状态字段建议统一枚举

## 1. parse_status / parse_jobs.status

- pending
- running
- success
- failed

## 2. tag link status

- pending
- confirmed
- rejected

## 3. review_tasks.status

- pending
- approved
- rejected
- fixed

## 4. files.upload_status

- pending
- uploaded
- confirmed
- failed

---

## 七、首月数据库实现建议

- 不要一开始追求高度范式化
- 不要拆太多附表
- 保证关键查询顺手
- 保证 AI 原始结果能回看
- 保证审核与追踪可落地

---

## 八、最终建议

> 这套数据库设计的核心不是“优雅”，而是让你在一个月内稳定支撑上传、解析、标签关联、图谱聚合、搜索和 AI 参考解析。
