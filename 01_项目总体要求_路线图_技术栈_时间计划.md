# 项目总体要求、路线图、技术栈与时间计划

## 一、结论先行

### 1. 一个月能否落地

**可以，但只能以“可用 MVP”口径落地。**

更准确地说：

- 如果你要的是“比赛可演示、可注册登录、可上传、可打标签、可关联、可检索、可展示基础知识图、可 AI 参考解析”的版本，**一个月可实现**
- 如果你要的是“成熟商业化 App + 多端完整打磨 + 强鲁棒性 + 强监控 + 完整题库体系 + 高精度知识图谱推理”，**一个月风险极高，不建议作为首月目标**

### 2. 为什么判断可行

因为本项目的关键能力并不是自己训练模型，而是把现成能力串起来：

- 前端负责采集与展示
- 对象存储负责图片/文件保存
- 服务端负责业务规则
- 大模型负责 OCR、结构化提取、标签建议、参考解析
- 数据库负责关系绑定与搜索
- 图谱只是“数据关系的可视化结果”，首月不需要上图数据库

### 3. 首月必须做减法

为了保证质量不下降，甚至提升成功率，建议做以下简化：

1. **不做真正图数据库**，先用关系型数据库 + 边表表达知识图
2. **不做复杂语音链路**，语音问答留到第二阶段
3. **不做 App 商店上架**，首月优先 H5/Web，可选再打 Android 包
4. **不做向量库强依赖**，首月先用全文检索 + 标签检索，向量召回作为增强
5. **不做重型微服务**，首月采用“逻辑分层、统一部署、接口先行”的方式

## 二、项目目标

## 1. 产品目标

做一个“学生自己的 AI 学习伴侣”：

- 上传错题图片或文字
- 上传笔记图片或文字
- 由 AI 自动提取题目/笔记内容与知识点
- 把“错题”和“笔记”用知识点自动连接
- 形成个人知识图
- 给出薄弱知识点
- 当本地题库没有命中时，由 LLM 给出**参考解析**

## 2. 首月 MVP 成功标准

满足下面 8 条即可判定 MVP 成功：

1. 用户可注册或登录
2. 用户可上传图片或文本
3. 文件可存入对象存储
4. 服务端可发起解析任务并拿到结构化结果
5. 错题/笔记可自动打标签，并支持人工修正
6. 笔记与错题能基于标签形成关联
7. Web 端可看到基础知识图或弱点图
8. 本地题库未命中时，可返回 AI 参考解析与置信度

## 3. 首月不做的内容

以下全部延期，不影响比赛 MVP：

- 教师端/班级端
- 社交功能
- 语音实时对话
- 真正复杂的学习路径推荐
- 图数据库推理引擎
- 多租户 SaaS 架构
- 灰度发布平台
- 自动计费/订阅系统

## 三、推荐技术栈（经过简化优化）

## 1. 前端

### 推荐方案

- **Uni-app + Vue 3 + TypeScript + Pinia**
- UI 组件：uni-ui 或轻量组件库
- Web 图谱：**ECharts 优先**
- 高级图编辑：G6 仅作为二阶段可选项

### 为什么这样选

- Uni-app 适合“一套代码覆盖 Web/H5/移动端”
- Vue 生态对 AI 生成代码较友好
- ECharts 上手快，比赛展示效果好
- G6 很强，但首月会增加交互与性能复杂度，建议后置

## 2. 服务端

### 推荐方案

- **Python 3.11 + FastAPI**
- ORM：SQLAlchemy
- 数据校验：Pydantic
- 后台任务：**数据库任务表 + 独立 worker 轮询**
- 不急着上 Celery、Kafka、K8s

### 为什么这样选

- FastAPI 适合快速构建接口
- Python 适合调用 LLM、做文本处理
- “任务表 + worker”比引入消息队列更容易落地和排障
- 对没有编程经验的项目负责人更好维护

## 3. 数据库

### 推荐方案

- **PostgreSQL**
- 首月使用：
  - 普通关系表
  - JSONB
  - Full Text Search
- 可选增强：
  - pgvector（第二阶段）

### 为什么推荐 PostgreSQL

因为它在一个库里就能兼顾：

- 常规业务关系
- JSON 结构存储
- 全文搜索
- 后续向量扩展

这样能减少系统组件数量。

## 4. 存储层

### 推荐方案

- 生产：**七牛云 Kodo**
- 本地开发：**MinIO**
- 统一抽象为 `StorageProvider`

### 为什么这样选

- 生产用对象存储，节省服务器磁盘压力
- 本地开发用 MinIO，方便模拟
- 统一接口后，存储与服务端可以并行开发

## 5. AI 能力层

### 推荐方案

- 主模型：Gemini 多模态模型
- 输出格式：强制 JSON Schema
- 任务：
  - OCR/图文理解
  - 错题结构化
  - 知识点候选提取
  - 标签置信度输出
  - 本地题库未命中时参考解析

### 原则

- **AI 只负责“建议与提取”，不直接当数据库真相**
- 真正写入知识标签前，要经过：
  1. 规范化映射
  2. 置信度判断
  3. 人工复核入口

## 6. 部署

### 首月推荐

- **一台 Linux 云主机 + Docker Compose**
- 组件：
  - Nginx
  - frontend
  - backend
  - worker
  - PostgreSQL
- 对象存储走外部服务

### 不建议首月做

- Kubernetes
- 多云部署
- 服务网格
- 复杂 CI/CD 平台

## 四、推荐总体架构

## 1. 简化后的首月架构

```text
[Uni-app Web/H5]
      |
      v
 [FastAPI API]
      |
      +--> PostgreSQL
      |
      +--> StorageProvider
      |      +--> MinIO（本地）
      |      +--> 七牛云（生产）
      |
      +--> Parse Job Table
      |
      +--> Worker
              |
              +--> Gemini API
```

## 2. 为什么不建议首月拆成重型微服务

你的想法“存储段与服务端分开开发，上游衔接，同时开发”是对的，但更适合这样理解：

- **开发边界分开**
- **接口契约先定**
- **部署先不强拆**

也就是说：

- 存储段可以独立开发
- 服务端可以独立开发
- 前端也可以并行开发
- 但首月上线时不一定要 3 套独立微服务

这样能减少：

- 认证复杂度
- 服务调用调试难度
- 部署成本
- 跨服务错误定位成本

## 五、并行开发策略

## 1. 可以并行的部分

### A. 存储段代理

负责：

- 文件上传策略
- 对象命名规范
- MinIO/Qiniu 适配器
- 文件元数据落库约定

### B. 服务端代理

负责：

- 用户、题目、笔记、标签、任务表
- 解析工作流
- AI 打标签
- 搜索与图谱聚合接口

### C. 前端代理

负责：

- 登录页
- 上传页
- 列表页
- 错题/笔记详情页
- 标签审核页
- 图谱页

### D. QA 代理

负责：

- 验收用例
- 回归测试清单
- 风险提示

### E. 运维代理

负责：

- Docker Compose
- 环境变量
- 本地启动说明
- 服务器部署说明

## 2. 并行开发的前提

必须先定 3 个契约：

1. **数据库核心表**
2. **API 接口契约**
3. **文件元数据结构**

只要这 3 个先定，多个代理就能并行。

## 六、30 天时间计划

## 第 1 周：定边界、跑通最小链路

目标：先把“上传 -> 落库 -> 可查看”跑通

交付：

- 项目目录结构
- PostgreSQL 初始化脚本
- 登录/鉴权基础能力
- 文件上传到 MinIO/Qiniu
- 文件确认接口
- 错题/笔记基础创建
- 前端基础页面骨架

第 1 周完成标准：

- 前端能上传一张图
- 后端能记录文件与条目
- 数据库能看到对应记录

## 第 2 周：跑通 AI 解析与标签链路

目标：实现“上传 -> AI 提取 -> 标签建议 -> 人工确认”

交付：

- parse_job 任务表
- worker 轮询任务
- Gemini 结构化输出 prompt
- 标签规范化逻辑
- 审核队列
- 前端审核页

第 2 周完成标准：

- 上传一张错题图后，系统能自动给出：
  - OCR 内容
  - 题目类型
  - 知识点候选
  - 置信度

## 第 3 周：做关联、搜索、图谱

目标：让错题与笔记真正“连起来”

交付：

- note_tag_links
- problem_tag_links
- knowledge_mastery 统计
- 搜索接口
- 知识图/弱点图接口
- Web 图谱页
- 错题详情一键查看相关笔记

第 3 周完成标准：

- 在某条错题详情里，能看到相关笔记
- 在图谱页能看到个人知识点节点

## 第 4 周：做本地题库未命中 fallback、测试、部署、演示

目标：让系统具备“没搜到也能给参考解析”的完整闭环

交付：

- 本地题库搜索逻辑
- LLM 自主解题 fallback
- AI 解析结果标注“参考答案”
- QA 回归
- Docker Compose 部署
- 演示数据准备
- 比赛演示脚本

第 4 周完成标准：

- 任意上传一题，若本地题库没命中，可返回 AI 参考解析
- 整体系统能在服务器部署并演示

## 七、核心风险与规避

## 1. 风险：范围膨胀

表现：

- 又想做移动端原生体验
- 又想做强图谱
- 又想做语音
- 又想做老师端

规避：

- 首月只做学生端 MVP
- 任何“新增功能”必须回答：它是不是比赛必须项？

## 2. 风险：AI 标签不稳定

表现：

- 同一题多次识别标签不一致
- 标签命名混乱

规避：

- 建立 canonical tags（规范标签表）
- AI 只提候选标签
- 服务端统一映射
- 低置信度必须进入人工审核

## 3. 风险：并行开发最后接不上

表现：

- 上传字段不一致
- 前端接口名和后端接口不一致
- 数据结构冲突

规避：

- 以 `11_数据库核心表设计草案.md` 和 `12_API接口契约草案.md` 为唯一标准
- 任何改动先改文档，再改代码

## 4. 风险：你自己不懂代码，不知道系统是不是做对了

规避：

你不看代码，只看 4 件事：

1. 有没有截图/页面
2. 有没有日志或接口返回
3. 能不能按测试清单走通
4. 代理是否说明“如何验收”

## 八、最终建议

### 推荐版本定义

**建议你把首月目标改成：**

> “星图智学 MVP 内测版：实现错题/笔记上传、AI 标签提取、知识点关联、基础知识图展示、本地题库检索与未命中时 AI 参考解析。”

### 推荐技术路线一句话总结

> 前端一套 Uni-app 代码做 H5/Web，服务端用 FastAPI，数据库用 PostgreSQL，对象存储用七牛云/MinIO，AI 解析用 Gemini，首月只做关系型知识图，不做复杂图数据库。

### 可行性判断

- **MVP 落地可行性：8/10**
- **完整商业化版本首月可行性：3/10**

所以答案不是“做不做得成”，而是“你必须把首月目标定义对”。
