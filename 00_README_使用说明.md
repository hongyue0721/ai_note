# 星图智学（StarGraph AI）项目文档包使用说明

这是一套给“AI 协同开发”直接使用的文档包，目标不是写一份好看的方案书，而是让多个代理拿到后能马上开工，并且让一个**没有编程能力的项目负责人**也能读得懂、管得住、验得了。

## 你先看什么

建议阅读顺序：

1. `01_项目总体要求_路线图_技术栈_时间计划.md`
2. `08_零编程负责人操作手册_代理分工建议.md`
3. `07_代理构建方案.md`
4. `11_数据库核心表设计草案.md`
5. `12_API接口契约草案.md`
6. 再把 02~06、09、10 分发给对应代理

## 这套文档解决什么问题

它重点解决 5 件事：

- 明确“一个月能不能落地”
- 把复杂项目压缩成可交付的 MVP
- 让存储段、服务端、前端并行开发
- 让“本地题库搜不到时”的 LLM 自主解题有清晰流程
- 让你作为非技术负责人也能分工、检查、推进

## 交付口径

本套文档默认的“落地”定义是：

> 30 天内做出一个可演示、可注册/登录、可上传、可识别、可打标签、可关联笔记与错题、可检索、可展示基础知识图、可在本地题库未命中时给出 AI 参考解析的 MVP 内测版本。

不把下面这些当成首月强制目标：

- iOS/Android 商店正式上架
- 完整商业级权限系统
- 全量语音助手
- 真正复杂的图数据库分析
- 高并发 SaaS 多租户
- 完整运维平台和监控大盘
- 极致 UI 打磨

## 推荐执行方式

- 你负责“范围、优先级、验收”
- 总指挥代理负责“分解任务、看依赖、控节奏”
- 其他代理负责“各自一块落地”
- 所有代理都必须只认这套文档，不得各自发明新目标

## 必须坚持的 4 条规则

1. **先做 MVP，再做增强版**
2. **先接口契约，再并行开发**
3. **首月逻辑分层，部署不要过度拆服务**
4. **所有输出都要带“验收方式”**

## 你现在可以直接做什么

把以下内容发给总指挥代理：

```text
请先完整阅读：
00_README_使用说明.md
01_项目总体要求_路线图_技术栈_时间计划.md
07_代理构建方案.md
11_数据库核心表设计草案.md
12_API接口契约草案.md

你的任务：
1. 把项目拆成 30 天倒排计划；
2. 明确今天可以并行开工的任务；
3. 给出存储段、服务端、前端、测试、部署 5 个代理各自今天的交付件；
4. 任何设计冲突时，优先选择更简单、更稳、更容易验收的方案。
```
