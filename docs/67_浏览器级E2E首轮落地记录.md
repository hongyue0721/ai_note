## 4. 覆盖内容

### 4.1 student 登录流

覆盖：

1. 打开 student 页面；
2. 填写 demo_user / user123456；
3. 点击“登录并拉取资料”；
4. 断言页面出现登录后的真实状态：
   - `demo@example.com · active`
   - hero 区块中当前账号变为 `demo_user`
   - “知识点可视化”区块存在

### 4.2 admin 登录流

覆盖：

1. 打开 admin 页面；
2. 填写 admin / admin123456；
3. 点击“登录后台并加载数据”；
4. 断言页面出现登录后的真实状态：
   - `.profile-card` 出现管理员信息
   - “监控总览”存在
   - monitor panel 中的“待执行任务”“待审核任务”存在

### 4.3 student 上传 + parse-job 流

覆盖：

1. 打开 student 页面并登录；
2. 选择本地测试文件；
3. 发起 `upload-policy -> upload-local -> confirm -> ingestions`；
4. 断言“最近任务”区块出现真实 parse job；
5. 断言页面状态中的文件记录变为“已生成”。

### 4.4 student 搜索流

覆盖：

1. 打开 student 页面并登录；
2. 在“5. 搜索”区块输入关键词；
3. 点击“执行搜索”；
4. 断言搜索结果列表出现至少一条真实结果；
5. 断言结果中包含目标题干文本。

### 4.5 student AI solve 流

覆盖：

1. 打开 student 页面并登录；
2. 在“6. AI 参考解析”区块提交题目；
3. 断言结果区出现：
   - `最终答案`
   - `模型`
   - `解题步骤`
   - `知识点`

## 6. 中途问题与修正

本轮第一次写的 student/admin 测试并未一次通过，主要原因不是产品坏掉，而是：

1. student 页面的 label / a11y 映射与 CSS class 断言方式不稳定；
2. admin 页面对“待审核任务”的文本匹配出现歧义，需要缩小到 monitor panel 范围内断言；
3. 新增 student-upload E2E 后，又真实暴露出两个问题：
   - student 前端 document 上传时错误发送 `entity_type=document`，与后端 ingestion schema 不兼容；
   - 远端 student/admin 前端源码缺少 `src/config.ts`，导致前端容器一重建就失败；

修正后，测试已稳定通过。
