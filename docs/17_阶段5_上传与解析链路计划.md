# 阶段 5：上传与解析链路计划

## 1. 目标

开始把“学生端真的能用”的业务主链路落到后端：

1. 文件记录
2. 内容采集入口
3. parse_jobs 建立与查询

## 2. 本阶段最小交付

### 文件链路

1. `POST /v1/files/upload-policy`
2. `POST /v1/files/confirm`

说明：

比赛级阶段可以先不接真实对象存储上传签名，先生成可用的占位 upload policy 和 object key，并把文件记录落库。

### 内容采集链路

1. `POST /v1/ingestions`

要求：

- 支持 `problem` / `note`
- 至少支持 `file_id` 或 `text_content` 二选一
- 创建后返回 `parse_job_id`

### 解析任务链路

1. `GET /v1/parse-jobs/{job_id}`
2. `POST /v1/parse-jobs/{job_id}/retry`

## 3. 当前策略

本阶段优先做“真实数据可走通”，暂不做完整 AI 解析。

所以当前可接受：

1. 先把 parse job 真正创建出来；
2. 先让查询与重试接口可用；
3. 先不接 Gemini 实际解析；
4. 后续再由 worker 接管真正执行。

## 4. 验收标准

1. 用户登录后可创建文件记录；
2. 可确认上传；
3. 可创建 ingestion；
4. ingestion 会生成 parse job；
5. parse job 可查询；
6. failed 状态任务可重试。
