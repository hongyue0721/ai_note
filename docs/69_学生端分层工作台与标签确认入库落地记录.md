# 学生端分层工作台与标签确认入库落地记录

## 1. 本轮完成内容

本轮已经把学生端从“单页堆叠演示页”推进为“登录层 + 工作台层”结构，并补齐了“AI 标签预览 -> 用户可修改 -> 确认后再入笔记记录”的真实闭环。

当前学生端已具备：

1. 登录页面单独一层；
2. 登录后进入工作台；
3. 工作台并列四个入口：
   - 上传笔记
   - 询问问题
   - 笔记管理
   - 搜索
4. 询问问题支持图片输入；
5. 上传笔记/图片可先拿到 AI 标签建议；
6. 用户可修改标签后确认入库；
7. 确认后自动切到“笔记管理”并显示新记录。

## 2. 后端新增能力

### 2.1 新增上传标签预览接口

- `POST /v1/preview/upload-tags`

作用：

1. 支持在正式入库前先做 AI 分类与知识点标签预览；
2. 文本文件直接使用文本内容分类；
3. 图片在有 OpenAI key 时会把图片一起送去做多模态分类。

### 2.2 新增笔记确认入库接口

- `POST /v1/notes/confirm`

作用：

1. 接收前端确认后的标题 / 分类 / 文本 / 标签；
2. 正式创建 note 记录；
3. 同时补一条成功 parse_job，使现有图谱/任务展示链路继续成立。

## 3. 这轮顺手修掉的问题

### 3.1 用户数据串视图问题

原来以下接口没有按当前登录用户过滤：

1. `/v1/notes`
2. `/v1/problems`
3. `/v1/search`
4. `/v1/graph/overview`
5. `/v1/graph/weak-tags`

这会导致一个用户看到别人的数据。

本轮已修正为按当前 user scope 过滤。

### 3.2 笔记确认入库 500 问题

新加的 `/v1/notes/confirm` 在第一次实测时出现 500，根因是：

> 预览 schema 中的 `knowledge_candidates` 是 preview schema 对象，直接塞给 `ClassificationResult` 会触发 Pydantic 校验错误。

本轮已修正为先转成 dict 再写入 `result_json`。
