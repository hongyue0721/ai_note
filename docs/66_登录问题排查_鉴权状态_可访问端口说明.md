# 登录问题排查、鉴权状态与可访问端口说明

## 1. 结论先说

当前“登录有问题”的根因已经定位清楚：

1. **后端登录接口本身是正常的**；
2. **用户 token / 管理员 token 解析也正常**；
3. 浏览器里表现成“登录有问题”的真实原因是：

> 远端运行时 `CORS_ORIGINS` 只允许 `localhost` 源，未允许当前真实前端来源 `http://146.190.84.189:3000` 与 `http://146.190.84.189:3001`。

所以：

- curl / 服务器本机调用能登录成功；
- 浏览器跨源预检会失败；
- 前端用户看到的就是“登录像坏了一样”。

本轮已经修复该问题。

## 2. 登录问题的实际证据

### 2.1 修复前的真实现象

在远端执行 preflight：

```text
OPTIONS /v1/auth/login
Origin: http://146.190.84.189:3000
=> 400 Disallowed CORS origin
```

管理员端同理：

```text
OPTIONS /v1/admin/auth/login
Origin: http://146.190.84.189:3001
=> 400 Disallowed CORS origin
```

### 2.2 修复前为什么 curl 登录还能成功

因为 curl 不受浏览器 CORS 预检限制，所以：

- `POST /v1/auth/login` 可以成功；
- `POST /v1/admin/auth/login` 也可以成功；

但这并不代表浏览器登录链路没问题。

## 3. 本轮修复内容

### 3.1 代码层

已更新：

- `apps/backend/app/core/config.py`

默认 `CORS_ORIGINS` 已补入：

- `http://localhost:3000`
- `http://localhost:5173`
- `http://127.0.0.1:3000`
- `http://127.0.0.1:3001`
- `http://146.190.84.189:3000`
- `http://146.190.84.189:3001`

### 3.2 运行时环境层

进一步确认真正决定线上行为的是：

- `/root/note/.env`
- `/root/note/apps/backend/.env`

二者修复前都只写着：

```text
CORS_ORIGINS=http://localhost:3000,http://localhost:5173
```

本轮已同步修正，并重建 backend 容器。

## 5. 鉴权这块是否已经做好

### 5.1 已经做好的部分

当前鉴权能力已经具备：

1. 用户注册/登录；
2. 用户 token 签发；
3. 用户 token 校验；
4. 管理员登录；
5. 管理员 token 签发；
6. 管理员 token 校验；
7. 用户/管理员作用域分离（`scope=user` / `scope=admin`）。

### 5.2 当前结论

因此更准确的说法是：

> **鉴权主链路已经做好并且可用，但前端登录体验、运行时 CORS 配置和前端分层仍有改进空间。**

换句话说：

- 不是“鉴权没做完”；
- 而是“鉴权已实现，浏览器接入层曾被 CORS 配置绊住”。
