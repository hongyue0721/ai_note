# 代理构建方案（适用于 AI 协同开发）

## 一、构建目标

本方案的目标不是“做很多 AI”，而是用最少的代理角色，把项目稳定推进到 MVP。

你需要的是：

- 分工清晰
- 依赖明确
- 文档统一
- 交付可验收
- 出问题时能快速定位

---

## 二、推荐代理拓扑

## 标准版：6 代理结构

1. **总指挥代理**
   - 负责排期、依赖、范围控制、验收把关

2. **存储段代理**
   - 负责对象存储、上传链路、文件元数据

3. **服务端代理**
   - 负责 API、数据库、AI 调用、搜索、图谱聚合

4. **前端代理**
   - 负责 Uni-app 页面与交互

5. **QA 测试代理**
   - 负责测试用例、回归清单、冒烟测试、问题归档

6. **部署运维代理**
   - 负责本地启动、Docker Compose、上线说明、环境变量

## 可选增强：第 7 个代理

7. **数据与 Prompt 代理**
   - 如果服务端压力过大，可单独负责：
     - 标签词表
     - Prompt 迭代
     - 测试样本集
     - 解析质量评估

---

## 三、为什么这样分工

这套分工不是按“技术名词”分，而是按**耦合边界**分：

- 存储段可以独立交付并 mock
- 服务端以数据库和 API 为核心
- 前端只依赖接口与状态
- QA 不写业务代码，专盯可用性
- 运维不介入业务判断，只负责能启动和能上线

这样最适合并行开发。

---

## 四、代理之间如何通信

## 1. 统一信息源

所有代理必须以以下文档为准：

- `01_项目总体要求_路线图_技术栈_时间计划.md`
- `11_数据库核心表设计草案.md`
- `12_API接口契约草案.md`

## 2. 统一改动规则

如果要改字段、状态、接口、目录结构：

- 先改文档
- 再改代码
- 再通知相关代理

禁止口头改约定。

## 3. 统一输出格式

每个代理每次输出必须包含：

1. 本次目标
2. 已完成内容
3. 变更文件
4. 如何运行/验证
5. 风险
6. 下一步

## 五、推荐项目目录结构

```text
stargraph-ai/
  docs/
  apps/
    frontend/
    backend/
    worker/
  libs/
    storage_adapter/
    shared_schemas/
  infra/
    docker/
    nginx/
    scripts/
  tests/
    api/
    e2e/
    fixtures/
```

### 说明

- `docs/`：放所有规范与任务文档
- `apps/frontend/`：Uni-app 前端
- `apps/backend/`：FastAPI 服务端
- `apps/worker/`：后台任务 worker
- `libs/storage_adapter/`：存储抽象及各 provider 实现
- `libs/shared_schemas/`：共享类型与枚举
- `infra/`：部署相关
- `tests/`：测试脚本与样例数据

## 六、推荐协作协议

## 1. 面向文档协作

每个代理开始工作前，都先读对应 md 文档。  
每次交付前，先回答：

- 我这次修改是否违背已有契约？
- 我是否影响别的代理？
- 我是否给了验收方式？

## 2. 面向契约开发

首月最重要的两个契约：

- 数据库表
- API 接口

如果契约不稳，整个多代理协作会混乱。

## 3. 面向验收交付

你要让每个代理都像外包团队一样交付，而不是像聊天室一样讨论。

---

## 七、建议的运行节奏

## 每天固定 3 个动作

### 早上

总指挥代理输出：

- 今日 3 个目标
- 各代理任务
- 今日验收标准

### 下午

各执行代理输出：

- 目前完成情况
- 阻塞
- 已生成的文件和接口

### 晚上

QA 或总指挥汇总：

- 今天是否达到验收标准
- 哪些内容要顺延
- 明天是否需要砍功能

---

## 八、失败恢复机制

如果某个代理产出很差或方向跑偏，使用以下方法：

1. 回到文档，重新对齐目标
2. 让总指挥代理重新拆任务
3. 缩小该代理任务粒度
4. 先要求它只交付一个最小可运行片段
5. 必要时由其他代理接手部分工作

---

## 九、推荐的“统一代理提示词模板”

你可以把下面这段给任意代理，再加上对应 md 文档：

```text
你是“星图智学（StarGraph AI）”项目中的一个专职代理。
请先完整阅读我提供的 md 文档，只根据文档目标工作，不要擅自扩展范围。

你每次输出都必须包含：
1. 你理解的目标；
2. 你本次要完成的最小交付件；
3. 你改动或新增的文件；
4. 如何让我一个不懂代码的人验收；
5. 当前风险；
6. 下一步。

如果出现设计冲突，优先选择更简单、更稳、更快可演示的方案。
```

## 十、推荐的“总指挥代理提示词模板”

```text
你是本项目总指挥代理。
请根据文档，把项目拆成 30 天计划，并给出今天各代理的任务。
要求：
- 以 MVP 为唯一目标；
- 先接口契约，后并行开发；
- 任何新增功能先判断是否首月必须；
- 如果存在风险，请主动砍掉非必要功能。
```

---

## 十一、你应该如何判断代理体系是否健康

如果出现下面这些现象，说明代理体系已经失控：

- 每个代理都在改别人的模块
- 同一个字段有多个命名
- 前端和后端互相等接口
- 代理交付不包含验收方式
- 总指挥代理只会讲空话，不会排依赖
- 每天做的事情很多，但没有一条完整链路跑通

健康状态的表现应该是：

- 第 1 周就能跑通上传链路
- 第 2 周就能看到 AI 解析结果
- 第 3 周能看到关联效果
- 第 4 周能稳定演示

---

## 十二、最终建议

> 这个项目不需要“很多代理”，而需要“少量代理 + 统一文档 + 强验收”。
> 只要你把代理边界和文档控制好，一个月做出比赛级 MVP 是现实的。
