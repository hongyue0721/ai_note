# 项目介绍摘要（新执行者必读）

## 1. 这是什么项目

这是一个面向学生的 **AI 学习整理系统**，当前项目名可理解为 **Smartnote / StarGraph AI**。

它不是普通网盘、也不是普通题库，而是一个把学生上传的学习材料转成“可整理、可搜索、可关联、可复盘”的系统。

核心处理对象包括：

- 错题图片
- 笔记图片
- PDF / doc / txt 等学习文档
- 文本输入的题目或笔记

---

## 2. 项目目标

当前目标不是商业级 SaaS，而是：

> **做一个比赛级完整成品**

这意味着：

- 核心功能要真实可用
- 学生端要能走完整主流程
- 后台管理端要能审核和监控
- 系统要能部署、能运行、能演示、能验收
- 但不追求首版就做到企业级高可用/高扩展

---

## 3. 当前核心业务流程

### 学生端主流程

1. 用户注册/登录
2. 上传图片、笔记或文档
3. 选择内容分类（problem / note / document）
4. 系统创建 ingestion 与 parse job
5. worker 执行解析/分类
6. 生成知识点候选与结构化结果
7. 低置信度内容进入审核队列
8. 学生可查看内容、状态、搜索结果和图谱概览
9. 后续还要支持本地题库未命中时的 AI 参考解析（solve）

### 后台端主流程

1. 管理员登录
2. 查看监控总览
3. 查看审核任务
4. 审核低置信度内容
5. 查看 parse jobs 列表和状态

---

## 4. 当前技术决策（已经定死，不要再改方向）

### 存储方案

当前固定为：

- **本地 PostgreSQL**：存结构化数据
- **本地文件存储**：存上传文件本体

不要再把方向改回：

- MinIO 优先
- 云对象存储优先
- SQLite 主存储

除非后续明确进入新阶段，否则继续按这个方案推进。

### 当前推荐本地路径

#### Windows
```text
D:\vibe_coding\note\data\uploads
```

#### Linux / 远端
```text
/root/note/data/uploads
```

#### 文件路径组织建议
```text
data/uploads/{user_id}/{content_category}/{yyyy}/{mm}/{uuid}-{filename}
```

### 数据库存什么

PostgreSQL 负责存：

- users
- admin_users
- files
- problems
- notes
- parse_jobs
- review_tasks
- 后续 AI solve 历史（建议继续补）

### 文件怎么存

文件本体：
- 存本地磁盘目录

数据库只存：
- `object_key`
- `file_url`
- `mime_type`
- `file_kind`
- `content_category`
- `size_bytes`
- `sha256`
- `upload_status`
- `storage_provider`

**不要把文件二进制塞进数据库。**

## 6. 当前还没完全完成的地方

### AI 真实能力还没彻底打通

虽然代码已经接好了：
- OpenAI client
- classifier service
- solver service
- `/v1/solve`

但**还缺真实 OpenAI 配置联调**。

也就是说：

- 有代码
- 有路由
- 有 prompt
- 但不一定已经在真实 key 下完成端到端验证

### 前端还不是最终完整产品页

当前前端是比赛级可用版，不是完整多页面商业产品版。

还没完全做满的地方：

- 更完整的详情页路由
- 更成熟的上传交互（拖拽/进度/预览）
- solve 结果前端展示收口
- 后台更完整筛选和操作体验

### 部署还差最后一公里

当前已补：
- compose
- frontend Dockerfile
- smoke 脚本

但还没完全做成：

- 远端 `docker compose up -d` 全链路实测固定化
- systemd / nginx / caddy 持久运行方案
