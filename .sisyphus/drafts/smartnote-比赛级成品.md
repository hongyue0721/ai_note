# Draft: Smartnote 比赛级成品

## Requirements (confirmed)
- 做一个能够运行的成品，保证所有功能都能够正常使用，即成品
- 目标选择：比赛级成品，但要求所有核心功能真实可用
- 新增要求：后端能够有监控功能
- 新增要求：需要有管理功能，即有管理界面
- 管理界面档位：核心后台
- 监控档位：基础监控
- 后台实现形式：独立后台前端
- 管理员账号体系：单独一套管理员账号体系
- 后台内容范围：只看解析任务和审核内容，不做完整内容总览

## Technical Decisions
- 产品级别：按“比赛级完整成品”规划，而不是正式商业产品级
- 基线范围：保留现有文档中的学生端核心链路（登录/上传/解析/标签/关联/搜索/图谱/AI 参考解析）
- 范围扩展：在学生端链路之外，补充管理界面与基础监控能力
- 前端形态：拆分为学生端前端 + 独立后台前端
- 管理员鉴权：与普通用户鉴权分离，单独管理员登录入口与账号体系
- 后台范围：解析任务状态/失败重试 + 审核内容处理；不做用户列表与内容总览后台
- 监控范围：服务健康、任务数量、成功/失败、最近错误日志、队列积压

## Research Findings
- `README.md`：当前仓库仍处于 MVP 骨架初始化阶段，真实业务接口与页面大多未完成
- `docs/00_开发进度与交接记录.md`：已完成目录骨架、最小 FastAPI 入口、worker 占位、Docker Compose 骨架
- `12_API接口契约草案.md`：已定义完整 API 契约，但当前代码仅实现极少数基础接口
- `05_前端代理_UniApp与知识图可视化说明.md`：已有标签审核页、状态页、图谱页等前端目标说明
- `10_部署运维代理_环境与上线说明.md`：强调可启动、可演示，但未形成完整监控平台或后台管理系统
- `04_服务端代理_API_AI任务与知识图谱说明.md`：已有 parse_jobs、审核队列、JWT/登录建议，可作为后台“任务 + 审核”能力基础
- `12_API接口契约草案.md`：已有 `/v1/review/tasks`、`/v1/parse-jobs/{job_id}`、重试接口、`/v1/dashboard` 草案，可扩展为后台/API 监控基础

## Scope Boundaries
- INCLUDE: 学生端核心链路真实可用、基础管理界面、基础监控能力、可部署可验收
- EXCLUDE: 商业级高可用、复杂告警平台、多租户、重型微服务、完整运营后台体系、完整内容运营后台、普通用户/管理员统一账号体系
